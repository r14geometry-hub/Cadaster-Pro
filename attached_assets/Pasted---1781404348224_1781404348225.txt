# ТЗ: Исправление прав доступа, блокировок пользователей и раздела «География»

## Цель

Обеспечить корректное разграничение прав между ролями:

* superadmin
* admin
* engineer
* customer

а также полностью запретить использование сервиса заблокированными пользователями.

---

# 1. Роли пользователей

## Superadmin

Имеет полный доступ ко всем функциям системы.

Может:

* видеть всех пользователей;
* видеть администраторов;
* видеть суперадминистраторов;
* блокировать и разблокировать:

  * admin;
  * engineer;
  * customer;
* управлять географией;
* управлять реквизитами;
* просматривать все данные системы.

Ограничения:

* не может заблокировать самого себя.

---

## Admin

Имеет ограниченный административный доступ.

Может:

* видеть только:

  * customer;
  * engineer;
* блокировать и разблокировать:

  * customer;
  * engineer;
* видеть раздел «География»;
* просматривать данные географии.

Не может:

* видеть пользователей с ролью admin;
* видеть пользователей с ролью superadmin;
* блокировать пользователей с ролью admin;
* блокировать пользователей с ролью superadmin;
* изменять географию;
* видеть раздел «Реквизиты»;
* получать данные раздела «Реквизиты» через API.

---

## Engineer

Может пользоваться только функционалом исполнителя.

---

## Customer

Может пользоваться только функционалом заказчика.

---

# 2. Защита на уровне Backend

Backend является основным уровнем защиты.

Frontend не считается защитой.

---

## GET /admin/users

Если текущий пользователь:

### superadmin

Возвращать всех пользователей.

### admin

Возвращать только:

* customer
* engineer

Не возвращать:

* admin
* superadmin

Скрытие должно работать независимо от frontend.

---

## PATCH /admin/users/:userId

Перед выполнением операции необходимо получать целевого пользователя.

Если:

```text
caller.role = admin
target.role = admin
```

вернуть:

```json
{
  "error": "Недостаточно прав"
}
```

HTTP 403.

---

Если:

```text
caller.role = admin
target.role = superadmin
```

вернуть:

```json
{
  "error": "Недостаточно прав"
}
```

HTTP 403.

---

Если:

```text
caller.id = target.id
isBlocked = true
```

вернуть:

```json
{
  "error": "Нельзя заблокировать самого себя"
}
```

HTTP 400.

---

# 3. Раздел «География»

## Видимость

Раздел должен отображаться:

* superadmin;
* admin.

---

## Режим superadmin

Полный доступ:

* просмотр;
* редактирование;
* изменение статусов;
* управление монетизацией;
* любые операции раздела.

---

## Режим admin

Только просмотр.

Доступно:

* просмотр субъектов РФ;
* просмотр статусов;
* просмотр справочников.

Недоступно:

* изменение данных;
* изменение статусов;
* изменение монетизации;
* любые операции редактирования.

---

## API

### Просмотр

```text
GET /admin/regions
```

Доступен:

* admin
* superadmin

---

### Изменение

```text
PATCH /admin/regions/:id
```

Доступен:

* только superadmin

Для admin возвращать:

HTTP 403.

---

# 4. Раздел «Реквизиты»

Раздел предназначен исключительно для superadmin.

---

## Superadmin

Видит раздел.

Имеет полный доступ.

---

## Admin

Не видит раздел.

Не получает раздел через API.

Попытка доступа:

HTTP 403.

---

# 5. Блокировка пользователей

## Основное правило

Заблокированный пользователь не должен пользоваться сервисом.

Независимо от роли:

* customer;
* engineer;
* admin;
* superadmin.

---

# 6. Middleware проверки блокировки

После успешной авторизации система обязана проверять:

```text
users.isBlocked
```

Если:

```text
isBlocked = true
```

вернуть:

HTTP 403

```json
{
  "error": "Вы заблокированы"
}
```

---

# 7. Ограничения для заблокированного пользователя

Запрещено:

* создавать заявки;
* редактировать заявки;
* откликаться на заявки;
* участвовать в чатах;
* отправлять сообщения;
* загружать файлы;
* подавать жалобы;
* получать контакты;
* пользоваться личным кабинетом;
* пользоваться админкой;
* пользоваться API платформы.

---

# 8. Интерфейс заблокированного пользователя

После входа пользователь должен видеть сообщение:

```text
Вы заблокированы.
Доступ к сервису ограничен.
```

---

Если API возвращает:

```json
{
  "error": "Вы заблокированы"
}
```

Frontend обязан показывать соответствующее сообщение.

---

# 9. Проверки после реализации

Необходимо подтвердить тестами.

---

## Проверка admin

Admin:

* не видит admin;
* не видит superadmin;
* не может блокировать admin;
* не может блокировать superadmin;
* видит раздел «География»;
* не может редактировать географию;
* не видит раздел «Реквизиты».

---

## Проверка superadmin

Superadmin:

* видит всех пользователей;
* может блокировать admin;
* может блокировать engineer;
* может блокировать customer;
* не может заблокировать самого себя;
* видит раздел «География»;
* видит раздел «Реквизиты».

---

## Проверка блокировок

Заблокированный customer:

* не может создать заявку.

Заблокированный engineer:

* не может откликнуться;
* не может писать сообщения.

Заблокированный admin:

* не может пользоваться админкой.

Для всех заблокированных пользователей:

```json
{
  "error": "Вы заблокированы"
}
```

HTTP 403.
