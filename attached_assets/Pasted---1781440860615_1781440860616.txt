# ТЗ: Полностью автоматическая проверка кадастровых инженеров

## Цель

Допускать к заявкам только действующих кадастровых инженеров, автоматически проверенных по аттестату и членству в СРО.

## Основной принцип

Роль engineer не назначается вручную.

Пользователь получает статус кадастрового инженера только после успешной автоматической проверки.

Админ и суперадмин не могут вручную сделать заказчика инженером.

---

## 1. Регистрация инженера

При регистрации инженер указывает:

* ФИО полностью;
* email;
* телефон;
* номер аттестата кадастрового инженера;
* регион деятельности;
* СРО, если известно.

После отправки формы система автоматически запускает проверку.

---

## 2. Что проверяет система

Система должна автоматически проверить:

1. Аттестат существует.
2. Аттестат действующий.
3. ФИО совпадает с данными реестра.
4. Инженер не исключён из реестра.
5. Инженер состоит в СРО.
6. СРО действующая.
7. Статус инженера не приостановлен и не аннулирован.

---

## 3. Статусы проверки

Добавить статусы:

* pending — проверка выполняется;
* verified — инженер подтверждён;
* rejected — данные не подтверждены;
* suspended — статус приостановлен;
* revoked — аттестат аннулирован;
* sro_invalid — нет действующего членства в СРО;
* verification_error — ошибка внешнего сервиса.

---

## 4. Успешная проверка

Если все данные подтверждены:

* user.role = engineer;
* engineer.isVerified = true;
* verificationStatus = verified;
* rosreestrStatus = active;
* sroStatus = active;
* записать дату проверки.

Инженер получает доступ:

* к личному кабинету инженера;
* к заявкам;
* к откликам;
* к покупке лидов;
* к отображению в каталоге.

---

## 5. Неуспешная проверка

Если аттестат не найден, ФИО не совпадает или СРО недействующая:

* user.role не становится engineer;
* engineer.isVerified = false;
* verificationStatus = rejected / sro_invalid / revoked / suspended;
* пользователь не отображается в каталоге;
* пользователь не может откликаться на заявки.

Показать сообщение:

"Данные кадастрового инженера не подтверждены. Проверьте ФИО, номер аттестата и членство в СРО."

---

## 6. Защита backend

Backend обязан проверять статус инженера перед каждым действием.

Если engineer.isVerified != true:

запретить:

* просмотр доступных заявок;
* отклик на заявки;
* покупку контактов;
* создание профиля инженера;
* отображение в каталоге;
* отправку сообщений заказчику как инженер.

Ответ API:

HTTP 403

{
"error": "Кадастровый инженер не подтвержден"
}

---

## 7. Админ-панель

В админке показывать раздел "Проверка инженеров".

Но админ не может вручную подтверждать инженера.

Админ видит только:

* ФИО;
* аттестат;
* СРО;
* статус автоматической проверки;
* дату последней проверки;
* причину отказа;
* кнопку "Повторить проверку".

Разрешённые действия:

* повторить автоматическую проверку;
* заблокировать пользователя;
* посмотреть историю проверок.

Запрещено:

* вручную подтвердить инженера;
* вручную изменить роль на engineer;
* вручную изменить verificationStatus на verified.

---

## 8. Повторная автоматическая проверка

Система должна автоматически перепроверять всех инженеров.

Периодичность:

1 раз в 7 дней.

Если при повторной проверке:

* аттестат аннулирован;
* инженер исключён из СРО;
* статус приостановлен;
* инженер не найден;

то:

* engineer.isVerified = false;
* verificationStatus = suspended / revoked / sro_invalid;
* инженер скрывается из каталога;
* инженер не получает новые заявки;
* отправляется уведомление.

---

## 9. История проверок

Создать таблицу verification_logs.

Поля:

* id;
* userId;
* engineerId;
* attestatNumber;
* checkedAt;
* source;
* result;
* status;
* sroName;
* sroStatus;
* errorMessage;
* rawResponse.

Каждая проверка должна сохраняться.

---

## 10. Источники данных

Реализовать архитектуру через провайдера:

RosreestrProvider

Методы:

* lookupByAttestat(attestatNumber)
* lookupByName(fullName)
* checkSroMembership(attestatNumber)
* normalizeAttestatNumber(attestatNumber)

На первом этапе можно оставить mock-провайдер только для разработки.

В production mock использовать запрещено.

Если production и реальный провайдер не настроен:

* регистрация инженера невозможна;
* показывать сообщение:
  "Автоматическая проверка временно недоступна. Попробуйте позже."

---

## 11. Интерфейс регистрации

После ввода аттестата:

Кнопка:

"Проверить аттестат"

Состояния:

* "Проверяем данные..."
* "Аттестат подтверждён"
* "СРО подтверждено"
* "Данные не подтверждены"
* "Проверка временно недоступна"

---

## 12. Карточка инженера

Показывать бейдж только для verified:

"Проверенный кадастровый инженер"

В карточке показывать:

* номер аттестата;
* СРО;
* дату последней проверки;
* регион деятельности.

---

## 13. Требования безопасности

Запрещено:

* назначать роль engineer вручную;
* подтверждать инженера через frontend;
* доверять данным из формы без проверки backend;
* использовать mock-реестр в production.

Все критические проверки должны быть на backend.

---

## 14. Критерии готовности

Функция считается готовой, если:

1. Пользователь без подтверждённого аттестата не может стать инженером.
2. Инженер с неподтверждённым статусом не видит заявки.
3. Инженер с неподтверждённым статусом не отображается в каталоге.
4. Админ не может вручную подтвердить инженера.
5. Суперадмин не может вручную подтвердить инженера.
6. Повторная проверка автоматически отключает инженера при проблемах с аттестатом или СРО.
7. Все проверки записываются в журнал.
8. В production mock-проверка отключена.
